## Context

Users organize items with tags and filter the list by tag (critical feature "organize my items";
acceptance: a tagged item is findable by its tag). Free tier caps tag count to drive upgrades.

## Data model

| Table       | Columns                                             | RLS                          |
| ----------- | --------------------------------------------------- | ---------------------------- |
| `tags`      | `id`, `owner_id`, `name` (unique per owner, citext) | owner-scoped, mirror `items` |
| `item_tags` | `item_id`, `tag_id` (PK pair)                       | via owning item              |

New migration only — existing migrations are immutable. Regenerate `database.types.ts` after.

## Flow

1. `/items` renders the tag picker (existing page, new `_components/tag-picker.tsx`).
2. `createTag` server action validates (`createTagSchema`), checks the free-tier limit, inserts.
3. `assignTag` validates ownership of both item and tag, upserts into `item_tags`.
4. Filtering happens server-side via a `tag` search param on the `/items` query.

## Edge cases & failure modes

- Duplicate tag name (case-insensitive) → surfaced as a form error, not a crash.
- Free-tier limit hit → upgrade prompt (same pattern as the item limit on `/items`).
- Tag deleted while assigned → `item_tags` rows cascade.
- No tags yet → picker shows its empty state, filter bar hidden.

## Out of scope

## Tag picker

Chip-style multi-select under the item list header. Uses the design-system chip/badge tokens
(`.impeccable/design.json`), keyboard navigable, `aria-pressed` on active chips. States: empty
(hidden), loading (skeleton row), error (inline retry).
